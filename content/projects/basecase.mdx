## Problem Definition

Most DSA practice trackers are flat lists. They don't model the hierarchical structure of how engineers actually study: organized sheets, grouped by topic sections, with individual problem-level tracking.

BaseCase solves this by treating DSA practice as a **relational data problem**.

## Architecture Overview

The system follows a standard full-stack architecture:

- **Frontend**: Next.js App Router with React Server Components
- **Backend**: Next.js API routes with Prisma ORM
- **Database**: PostgreSQL (hosted on Neon)
- **Deployment**: Vercel

## Data Modeling

The core schema follows a strict hierarchy:

### Sheet → Section → Problem

- A **Sheet** represents a study plan (e.g., "Striver's SDE Sheet")
- A **Section** groups problems by topic (e.g., "Arrays", "Dynamic Programming")
- A **Problem** is an individual DSA problem with status tracking

This relational design ensures:

1. **No orphaned data** — foreign keys enforce hierarchy
2. **Efficient reads** — query any level of the hierarchy independently
3. **Atomic writes** — nested transactions for bulk operations

## Nested Transaction Handling

The most complex operation is **bulk status updates**. When marking a section complete:

1. All problems in the section update to "completed"
2. Section progress recalculates to 100%
3. Parent sheet progress recalculates

This requires a database transaction to maintain consistency:

```typescript
await prisma.$transaction(async (tx) => {
  // Step 1: Update all problems
  await tx.problem.updateMany({
    where: { sectionId },
    data: { status: "completed" },
  });

  // Step 2: Recalculate section
  const [total, done] = await Promise.all([
    tx.problem.count({ where: { sectionId } }),
    tx.problem.count({ where: { sectionId, status: "completed" } }),
  ]);

  await tx.section.update({
    where: { id: sectionId },
    data: { progress: done / total },
  });
});
```

## State Synchronization

The frontend uses optimistic updates for responsiveness:

1. User clicks "Mark Complete"
2. UI immediately reflects the change
3. API call fires in background
4. On failure: UI reverts, error toast shown

This pattern avoids blocking the UI on network latency while maintaining data integrity.

## Scaling Considerations

- **Query optimization**: Added composite indices on `(sectionId, status)` for progress calculations
- **Connection pooling**: Prisma Accelerate for serverless connection management
- **Caching strategy**: Static generation for sheet listings, dynamic for progress data
- **Future**: Migrate to ISR when user count grows beyond single-user scope

## Lessons Learned

1. **Model the domain first.** The UI is a projection of the data model — get the schema right and everything else follows.
2. **Transactions are non-negotiable.** Multi-table writes without transactions will eventually corrupt state.
3. **Deploy to production early.** Connection pooling, cold starts, and migration ordering are invisible in local development.
4. **Measure before optimizing.** The initial simple queries were fast enough. Indices were added only after profiling.
